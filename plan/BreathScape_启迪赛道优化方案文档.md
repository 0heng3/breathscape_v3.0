# 息境 BreathScape 启迪赛道优化方案文档

## 面向 6–12 岁儿童的 AI 声景线描情绪场景日记

---

## 0. 文档目的

本文档用于重新整理“息境 BreathScape”项目的核心方向、当前实现不足、优化后的设计目标与精细化实现路径。文档重点面向中国移动应用创新大赛启迪赛道的参赛语境，强调：

- 项目应聚焦“小而美”的核心体验，而不是功能堆叠；
- 当前网页 Demo 存在儿童 App 质感不足、场景反馈弱、技术架构过浅等问题；
- 后续应围绕“绘本式场景 + 声景线描 + AI 温和共创 + 情绪场景日记”进行重构；
- 初赛阶段可用 Web 原型展示概念，复赛/决赛阶段应逐步转向 iPadOS / iOS / visionOS 兼容路线。

---

## 1. 项目定位重新定义

“息境 BreathScape”是一款面向 6–12 岁儿童的情绪表达与轻度调节交互原型。

项目不定位为心理诊断工具，也不替代专业心理咨询，而是借鉴艺术疗愈中的非语言表达、象征转化、过程安抚和作品回看机制，为儿童提供一个低门槛、非评判、可持续的情绪表达空间。

项目的核心不是让 AI 判断儿童“是什么情绪”，而是让儿童通过线条、声音和小场景，把说不清的心情转化为可以观看、可以听见、可以保存的情绪场景日记。

### 1.1 项目一句话

**息境 BreathScape 让孩子在一个会回应的小场景中，用线条画下今天的风、雨、草和光，把说不清的心情变成可以观看、可以听见、可以保存的情绪日记。**

### 1.2 第一阶段最小闭环

第一阶段不做大而全的平台，而聚焦一个小而完整的体验闭环：

```text
儿童选择今日小状态
→ AI 给出温和创作邀请
→ 儿童在花园场景中绘制元素
→ 场景以声音、动画和生长反馈回应儿童
→ 收尾安放
→ 生成今日情绪场景日记
```

第一版 Demo 以“花园”作为示例场景。花园不是唯一场景，而是用于验证“情绪场景共创”的代表性样例。后续可以扩展为海洋、星空等不同场景，但第一阶段必须先把花园做扎实。

---

## 2. 参赛背景与启迪赛道适配

启迪赛道强调原创性应用程序、设计创新、应用前景、功能性、可靠性与流畅性。结合赛道要求，本项目不应只停留在“网页展示页面”，而应逐步形成面向 iPadOS / iOS / visionOS 的应用原型路线。

### 2.1 当前阶段建议

当前阶段可在 Windows 环境下使用 Web 技术完成高质量交互原型，用于：

- 给导师演示核心机制；
- 生成初赛说明文档、效果图与录屏；
- 验证花园场景、元素反馈和情绪日记闭环。

但需要明确：Web Demo 是阶段性原型，不是最终参赛 App 形态。

### 2.2 后续路线建议

建议采用：

```text
Web prototype now → iPadOS App later → visionOS extension later
```

即：

- 初赛阶段：Windows + Web Canvas 原型；
- 复赛阶段：iPadOS WebView 包装或 SwiftUI 原生版本；
- 决赛阶段：提交可在 iPadOS 设备运行的 App；
- 长期拓展：visionOS 空间情绪场景作为未来亮点，而非当前主方案。

### 2.3 为什么不优先做 VR / visionOS

VR / visionOS 能提升想象力，但当前不适合作为主线，原因包括：

- 6–12 岁儿童使用头显存在设备可及性与安全性问题；
- Vision Pro 开发通常需要 Apple Silicon Mac、Xcode、visionOS SDK；
- 当前 2D 花园场景的儿童视觉质感和交互反馈尚未打透；
- 直接上 VR 会显著增加功能性、可靠性和流畅性风险。

更稳妥的做法是：

**主方案聚焦 iPadOS 儿童情绪场景日记，visionOS 作为长期拓展。**

## 3. 当前实现的主要不足

### 3.1 当前页面仍是“功能原型”，不是“儿童 App 场景体验”

目前 Demo 已经具备基础流程，包括首页、心情选择、AI 引导、绘画、呼吸收尾和日记卡。但整体呈现仍然偏网页式和说明型，像一个温柔的功能原型，而不是儿童真正愿意进入的 App 世界。

具体表现为：

- 页面多采用“左侧插图 + 右侧卡片”的结构；
- 信息表达清楚，但儿童沉浸感不足；
- 用户看到的是页面、按钮和卡片，而不是一个等待他进入、触摸和改变的小世界。

### 优化方向

将页面从“卡片式信息布局”改为“场景式体验布局”。花园不再只是插图背景，而应成为界面主体。

每一页都应围绕花园状态展开：

- 花园醒来；
- 花园等待；
- 花园回应；
- 花园安放；
- 花园被保存。

---

### 3.2 花园视觉识别度不足

当前画面中的花园偏抽象，缺少明确可识别的儿童视觉元素。土地、种子、草、花、小灯、天空、远景等没有形成完整的绘本式场景，因此儿童很难直观理解“这里是一个可以被我改变的小花园”。

### 优化方向

第一版花园需要具备明确视觉结构：

- 天空；
- 土地；
- 种子；
- 小灯；
- 少量小草；
- 柔和远景；
- 可互动区域。

画面可以简洁，但不能空泛。儿童应当一眼看出：

**这是一块还没完全长好的地方，我可以给它添一点东西。**

---

### 3.3 缺少稳定的情感锚点

当前系统中“AI 引导”主要通过文字卡片呈现，缺少一个贯穿全流程的陪伴角色。成熟儿童 App 往往有一个稳定的情感锚点，使孩子知道“谁在陪我”“谁在回应我”。

### 优化方向

引入“小灯”作为 AI 的温和化身。小灯不是普通装饰角色，而是承担引导、回应、收尾和日记旁白的场景生命。

小灯在不同阶段有不同状态：

| 阶段 | 小灯状态 | 功能 |
|---|---|---|
| 首页 | 睡在种子旁边，微弱发光 | 建立安全感 |
| 状态选择页 | 轻轻亮起 | 邀请孩子选择今日状态 |
| AI 引导页 | 亮度增强，说出建议 | 作为 AI 共创伙伴 |
| 绘画页 | 根据元素变化回应 | 体现花园在听孩子 |
| 收尾页 | 发出柔和光晕 | 帮助场景安放 |
| 日记页 | 在日记卡上留下小印章 | 完成仪式感 |

这样 AI 不再以“技术工具”出现，而是以场景中的陪伴者出现。

---

### 3.4 元素绘制后的场景反馈不够明显

目前绘画页容易出现“儿童线条叠在画布上”的效果，场景并没有被明显改变。这样会削弱核心交互：孩子画了雨，花园到底发生了什么？画了光，花园如何回应？

### 优化方向

必须区分“儿童笔迹层”和“系统回应层”。

- 儿童笔迹层：保留孩子的原始线条，作为情绪表达痕迹；
- 系统回应层：根据元素类型生成统一风格的雨滴、湿土、草芽、花苞、光晕、风痕等视觉反馈；
- 场景状态层：进一步改变整个花园的环境，例如土壤湿润、光线变暖、草地生长、云朵移动。

核心原则：

**儿童画的是线索，系统生成的是回应。**

---

### 3.5 当前纯 HTML 页面难以承载目标效果

当前如果仅依赖 HTML/CSS 和简单 JS，可以展示流程，但难以实现儿童 App 所需的场景变化、动态反馈、笔触分析、声音映射和长期状态管理。

### 优化方向

继续保留 HTML/CSS 作为页面外壳，但核心交互必须引入：

- Canvas / SVG 场景渲染；
- JavaScript 状态引擎；
- Web Audio / Tone.js 声音系统；
- 统一插画资产库；
- 小灯角色动效系统。

当前不必立即做原生 Swift App，但要按未来 iPadOS 迁移的结构设计 Demo。

## 4. 优化后的核心概念

### 4.1 艺术形式：声景线描

项目的艺术形式不再表述为普通绘画、水墨或卡通，而定义为：

**声景线描。**

声景线描指儿童用简单线条绘制场景元素，系统将这些线条转化为声音、动画和场景状态变化。

儿童不需要画得像，也不需要完成标准作品。风可以是一条波浪线，雨可以是几点短线，草可以是几根向上的线，光可以是几条放射线。

视觉风格采用：

- 儿童手绘线条；
- 轻水彩质感；
- 纸张肌理；
- 低饱和自然色；
- 柔和光影；
- 适度留白。

这种风格的目标不是生成完美画面，而是让孩子的线条能够自然融入场景，并被系统温和地接住。

---

### 4.2 情绪与元素的关系：从“情绪标签”转向“调节动作”

项目不采用“难过 = 雨”“烦躁 = 风”的直接映射。这样的逻辑过于直给，也缺少充分依据。

优化后的映射逻辑为：

```text
儿童状态 → 调节需求 → 场景元素
```

儿童入口不问“你焦虑吗”，而问：

**今天心里更像哪一种小状态？**

可选状态包括：

- 有点快；
- 有点重；
- 有点乱；
- 有点空；
- 有点亮；
- 说不清。

这些不是诊断标签，而是儿童可以理解的身体感受和场景感受。

系统内部将这些状态转译为调节需求：

| 儿童状态 | 调节需求 |
|---|---|
| 有点快 | 释放、放缓 |
| 有点重 | 被接住、被陪伴 |
| 有点乱 | 整理、稳定 |
| 有点空 | 滋养、轻微激活 |
| 有点亮 | 扩展、分享 |
| 说不清 | 允许模糊、不强迫命名 |

第一版保留五个基础元素：

| 元素 | 调节动作 |
|---|---|
| 风 | 释放与流动 |
| 雨 | 冲刷与滋养 |
| 草 | 恢复与韧性 |
| 花 | 打开与舒展 |
| 光 | 支持与温暖 |

元素不是情绪标签，而是儿童可以执行的调节动作。

## 5. 第一版 Demo 精细化页面方案

### 5.1 页面 1：首页：小花园醒来

#### 目标

建立世界感和安全感。

#### 页面结构

页面不再采用左右分栏的信息式布局，而是呈现一个完整的小花园入口。画面中有浅色天空、一小块土地、一颗小种子、一盏睡着的小灯和少量尚未长开的草芽。

#### 文案

```text
今天的小花园醒来了。
你不用马上说清楚心情，
可以先从一条线、一点雨、一束光开始。
```

#### 主按钮

```text
开始今天的花园
```

#### 视觉要求

- 场景占屏幕主体；
- 按钮和文字作为轻量叠加；
- 小灯微弱发光，提示这是一个会回应的空间；
- 页面应像绘本第一页，而不是网页欢迎页。

---

### 5.2 页面 2：今日状态选择

#### 目标

用儿童可理解的方式进入情绪表达。

#### 页面结构

当前的选项卡应从普通卡片升级为“小状态物件”。每个状态都用一个小插图表达，而不是仅靠文字。

#### 建议选项

| 状态 | 小插图表达 |
|---|---|
| 有点快 | 一条跑得很快的小路 |
| 有点重 | 一块压在土地上的小石头 |
| 有点乱 | 被风吹散的小叶子 |
| 有点空 | 一块还没长东西的土地 |
| 有点亮 | 一束小光 |
| 说不清 | 一片淡淡的雾 |

#### 页面文案

```text
今天心里更像哪一种小状态？
```

#### 补充文案

```text
说不清也没关系，小花园可以慢慢听。
```

#### 优化重点

不要让页面像测评问卷。状态选择应像孩子在花园中选择一个“今天的小物件”。

---

### 5.3 页面 3：AI 温和引导

#### 目标

体现 AI 作用，但不暴露为诊断系统。

#### 页面结构

AI 不显示为“正在分析情绪”，而由小灯在花园中说话。根据儿童选择的状态，小灯给出创作邀请。

#### 示例文案

如果选择“有点快”：

```text
花园里的风好像跑得有点快。
你可以先画一阵风，让它流动出来；
也可以画一点云，让它慢下来。
```

如果选择“有点空”：

```text
今天的土地还很安静。
不用急着开满花，
可以先画一点草，或者给它一点光。
```

#### 优化重点

AI 是“共创伙伴”，不是“情绪判定器”。所有提示都应是邀请式，而不是指令式。

---

### 5.4 页面 4：场景共创页

#### 目标

让儿童真正感到：

**我画的东西让花园发生了变化。**

#### 页面结构

画布即花园，不再是空白画板。底部提供五个元素入口：

- 风；
- 雨；
- 草；
- 花；
- 光。

儿童先选择元素，再在场景中画线。系统根据当前元素模式识别线条参数，不要求完全自动识别儿童画的内容。

这样既保证技术稳定，也减少儿童挫败。

---

#### 元素一：风

儿童输入：波浪线、长线、旋转线。

声音反馈：轻风声、风铃声；速度越快，风声越密。

场景反馈：草和云轻轻摆动，出现风线痕迹。

系统文案：

```text
风来了，它把花园里的声音轻轻带动了。
```

---

#### 元素二：雨

儿童输入：点状线、短线、下落线。

声音反馈：雨滴声；点越密，雨声越密。

场景反馈：土地颜色变深，出现湿润纹理，种子周围有水珠，小草生长概率增加。

系统文案：

```text
雨落下来了，土地慢慢喝到水了。
```

---

#### 元素三：草

儿童输入：向上的短线。

声音反馈：轻微沙沙声。

场景反馈：地面长出小芽，草地沿着土地边缘扩展。

系统文案：

```text
一点绿色长出来了。它不用很快，只要慢慢长。
```

---

#### 元素四：花

儿童输入：弧线、圆形、点状花瓣。

声音反馈：柔和开花音。

场景反馈：花苞从草地中出现，并根据光照和生长值逐步打开。

系统文案：

```text
花开了一点点。它不用一下子完全打开。
```

---

#### 元素五：光

儿童输入：放射线或轻点。

声音反馈：铃声、光点声。

场景反馈：画面亮度提升，色温变暖，小灯更亮，花苞更容易打开。

系统文案：

```text
光落在花园里，刚刚好。
```

---

#### 儿童端信息原则

右侧数字统计面板应删除或隐藏，不出现在儿童端。儿童端只看见花园状态变化。技术数据可放在汇报材料里解释，不应打断儿童体验。

---

### 5.5 页面 5：花园安放模式

#### 目标

完成从表达到收束的体验。

#### 页面文案

```text
现在，我们让花园慢慢停下来。
花园会把今天的线条轻轻收好。
```

#### 交互方式

儿童可以跟随光点进行 2–3 次慢呼吸，或长按屏幕让花园进入安放状态。

#### 视觉变化

- 儿童原始线条透明度降低但不消失；
- 过密线条被柔化为风痕、云雾、草丛或雨幕；
- 整体饱和度降低；
- 纸张肌理增强；
- 小灯亮起；
- 声音逐渐变低；
- 花园稳定在一个温和画面中。

#### 优化重点

不要强行把画面变成“漂亮治愈风”。如果孩子今天画得密集、凌乱，也应保留痕迹，只是被花园温和地收住。

---

### 5.6 页面 6：今日情绪场景日记

#### 目标

让一次互动沉淀为可回看的作品。

#### 日记卡内容

- 今日花园画面；
- 今日添加的元素；
- 一句小灯生成的温和故事回顾；
- 可选标题；
- 保存按钮。

#### 示例文案

```text
今天你先给花园带来了一阵风，
后来又画了一点小草。
风慢慢轻了一些，土地也长出了一点绿色。
这是今天的花园，也是你今天留下的一段心情。
```

#### 边界要求

日记卡不出现：

- 焦虑下降；
- 情绪改善明显；
- 心理风险提示；
- 心理评分；
- 诊断结论。

## 6. 技术实现方案

### 6.1 当前架构判断

当前纯 HTML/CSS 可以作为页面外壳，但无法独立支撑目标效果。原因是目标效果需要实时场景变化、绘画输入、音效映射、角色状态和图层渲染。

因此建议采用：

| 模块 | 技术 |
|---|---|
| 页面框架与基础布局 | HTML / CSS |
| 花园场景渲染和儿童绘制 | Canvas / SVG |
| 场景状态引擎与交互逻辑 | JavaScript |
| 声音反馈 | Web Audio / Tone.js |
| 本地日记保存 | LocalStorage / IndexedDB |
| 第一阶段 AI 引导 | 规则模拟 AI |
| 后续故事增强 | 可接入 LLM |

---

### 6.2 场景状态引擎

花园需要有内部状态，而不是固定插图。

基础状态变量包括：

```js
const gardenState = {
  soilMoisture: 0, // 土壤湿度
  lightLevel: 0,   // 光照强度
  windLevel: 0,    // 风力强度
  growthLevel: 0,  // 生长程度
  bloomLevel: 0,   // 开花程度
  calmLevel: 0,    // 平静程度
  densityLevel: 0  // 画面密度
};
```

不同元素改变不同状态：

| 元素 | 状态变化 |
|---|---|
| 雨 | soilMoisture + 20, growthLevel + 5 |
| 光 | lightLevel + 20, bloomLevel + 10 |
| 风 | windLevel + 20 |
| 草 | growthLevel + 15 |
| 花 | bloomLevel + 20 |

场景根据状态变化自动渲染：

- 土壤湿度高 → 土地变深，出现水珠；
- 光照高 → 画面变暖，小灯更亮；
- 风力高 → 草和云摆动；
- 生长值高 → 小草增多；
- 开花值高 → 花苞打开。

---

### 6.3 图层系统

画面需要分层管理：

| 图层 | 内容 |
|---|---|
| 背景层 | 天空、远景、土地 |
| 状态层 | 湿土、光晕、风、云 |
| 系统回应层 | 雨滴、小草、花、光点 |
| 儿童笔迹层 | 原始线条 |
| 柔化层 | 纸张肌理、水彩晕染、安放滤镜 |
| 角色层 | 小灯 |
| UI 层 | 按钮、元素入口、提示气泡 |

有了图层系统，儿童画得不好也不会破坏整体视觉。

---

### 6.4 笔触参数识别

系统需要记录：

- 笔触速度；
- 方向；
- 长度；
- 密度；
- 停顿时间；
- 绘制区域；
- 当前元素模式。

第一版不需要复杂机器学习。元素类型由儿童先选择，系统只分析同一元素下的笔触差异。

例如：

- 同样是雨，点得快则雨声更密；点得慢则雨声更轻；
- 同样是风，线条快则风更明显；线条慢则风更柔；
- 同样是光，轻点产生小光点，长线产生光束。

---

### 6.5 声音系统

声音是“声景线描”的关键，不应只是点击播放固定音效。

基础规则：

| 参数 | 声音影响 |
|---|---|
| 元素类型 | 决定音色 |
| 笔触速度 | 影响节奏 |
| 笔触密度 | 影响声音密度 |
| 收尾阶段 | 音量逐渐降低 |
| 快速乱画 | 自动限制音量和高频 |

第一版至少实现：

- 雨声；
- 风声；
- 草声；
- 开花音；
- 光点铃声；
- 收尾环境音。

## 7. 符合启迪赛道的技术路线

启迪赛道要求作品面向 iOS / iPadOS / visionOS 设计开发，并强调原创性、功能性、可靠性和流畅性。因此，当前 Web Demo 不能被定义为最终作品，只能作为阶段性原型。

### 建议路线

```text
初赛阶段：Windows 环境下完成高质量 Web 交互原型、录屏、效果图和方案文档。
复赛阶段：迁移为 iPadOS 准原生或原生版本。
决赛阶段：提交可在 iPadOS 设备中运行的 App。
长期拓展：visionOS 空间情绪场景作为未来亮点，而非当前主方案。
```

### 技术策略

| 阶段 | 技术策略 |
|---|---|
| 当前 | Web Canvas 原型 |
| 中期 | iPadOS WebView 包装或 SwiftUI 重构 |
| 长期 | SwiftUI + PencilKit + AVAudioEngine + Core ML |
| 拓展 | visionOS 空间情绪场景 |

主线应坚持：

**iPadOS-first，visionOS-later。**

不要现在直接主打 VR，否则开发风险和设备门槛过高。

---

## 8. 后续场景扩展原则

第一版只做花园。花园的情绪调节侧重点是恢复、照料、生长和慢慢打开。

后续如果扩展场景，不能只是换皮，必须对应不同情绪调节需求。

| 场景 | 情绪调节侧重点 | 适合状态 |
|---|---|---|
| 花园 | 恢复、照料、生长 | 低落、疲惫、没力气、需要重新开始 |
| 海洋 | 流动、释放、放缓 | 心里乱、烦躁、停不下来、需要疏导 |
| 星空 | 距离、想象、安静 | 说不清、睡前胡思乱想、需要放远一点看情绪 |

森林暂不作为优先场景。因为森林和花园同属自然植物意象，容易重复。除非后续明确把森林定义为“安全庇护”场景，否则不应与花园同时开发。

## 10. 优化后的评估目标

第一版 Demo 不证明临床疗效，而验证以下问题：

- 儿童是否能理解“今日小状态”的入口；
- 儿童是否愿意在场景中画元素；
- 儿童是否能感受到自己的绘制让花园发生变化；
- 儿童是否理解声音与线条之间的关系；
- 儿童是否愿意保存今日场景；
- 儿童是否能用简单语言描述“这个花园哪里像今天的心情”。

### 建议评估变量

| 层级 | 变量 | 测量方式 |
|---|---|---|
| 主要变量 | 即时情绪调节效果 | 使用前后儿童自评的舒适感、平静感或情绪天气变化 |
| 次要变量 | 情绪表达流畅度 | 是否愿意画、画了多少、是否能给作品命名 |
| 次要变量 | 情绪觉察能力 | 是否能说出某个元素与自己感受的关系 |
| 次要变量 | 自我安抚效能感 | 是否觉得自己可以通过画和呼吸让花园慢下来 |

---

## 11. 最终优化目标

优化后的 BreathScape 不应是一个“功能很多的儿童心理 App”，而应是一个“小而完整的情绪艺术体验”。

它的目标不是让 AI 告诉孩子怎么了，而是让孩子通过一个会回应的小场景，发现：

```text
我可以不用马上说清楚。
我可以先画一阵风、一点雨、一束光。
我的线条会被花园接住。
我的心情可以被看见、被听见，也可以被慢慢安放。
```

这才是项目最核心的价值。
